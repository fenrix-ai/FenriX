⚠️ Things that will trip you up

Firestore listener cleanup. If you don't unsubscribe from onSnapshot when components unmount, you'll get memory leaks and stale updates. Use useEffect cleanup functions religiously.

Budget validation. Do it client-side for instant feedback AND server-side for security. Never trust the client alone — students will find ways to submit invalid states.

Timer sync. Don't rely on client-side timers alone. The server-side game state should have a round_end_time timestamp. Client computes remaining time from that. Otherwise timers will drift across players.

Write rules so: players can only read/write their own player document, game state is read-only for players (only admin/Cloud Functions can write), leaderboard is read-only.
Without security rules, any student can open the browser console and edit anyone's data. This is not optional. Test rules with the Firebase emulator.

Gate these behind an admin check: verify the caller's UID matches a hardcoded professor UID, or use a custom claim. Don't let students trigger phase transitions.

Decision Handling & Revenue Engine
The heart of the game — this is the hardest part
Apr 8–14
Accept & validate player decisions
When a player submits, validate: budget isn't exceeded, prices are positive, staff count ≥ 0, ad_spend ≥ 0, ad_type is a valid category. Write validated decisions to the player's round sub-document.
Validate server-side in a Cloud Function, not just in Firestore rules. Rules can't do complex math (e.g., staffing_cost + ad_spend ≤ budget). A Cloud Function onWrite trigger works well here.
👤 Daniel + Scott
Revenue engine — regression model
Cloud Function triggered when all players have submitted (or timer expires). For each player: build feature vector from decisions → apply regression formula → compute revenue. Store results.
Formula: revenue = 500 + (30 × staff_count) − (15 × avg_price) + (0.8 × ad_spend) + (50 × num_products) + (5 × headchef_skill) + ad_type_bonus + noise. Load coefficients from a config doc so Game Design can tune without code changes.
⛓️ Needs final coefficients from Game Design
👤 Daniel
Customer allocation logic
Finite customer pool. Compute each café's attractiveness score (function of prices, products, staff, ads). Distribute customers proportionally. Write customer_count and customer_satisfaction to each player's round result.
⛓️ Needs customer allocation formula from Game Design
👤 Daniel
Per-product sales computation
Based on customer count and product mix, determine how many of each product were sold: croissant, cookie, bagel, sandwich, latte, matcha_latte. Write as integers to the round results.
👤 Scott
Round results assembly
After simulation, assemble the complete results document for each player: revenue, customer_count, customer_satisfaction, per-product sales, headchef_skill, ad_type. Write to /games/{gameId}/players/{playerId}/rounds/{roundId}.
👤 Daniel
Default handling for non-submitters
If a player doesn't submit before the timer: use their previous round's decisions as defaults (or a hardcoded baseline for round 1). Don't block the simulation waiting for stragglers.
👤 Scott
🚩

Leaderboard & Data Export
Students need to see rankings and download their data
Apr 14–17
Leaderboard computation
After each round simulation, aggregate cumulative revenue for all players. Rank them. Write to /games/{gameId}/leaderboard as a sorted array. Include rank change from previous round.
👤 Daniel
Time-series dataset storage
Each player accumulates round-over-round data. Store as an array of row objects in the player document, or as sub-documents under /players/{id}/data/{roundId}. Whichever is easier to query for CSV export.
Consider read costs. If Frontend fetches the entire dataset each time, that's 17 fields × N rounds × 1 read per field. Batch into a single document if possible to reduce reads.

⚠️ Things that will trip you up

Free tier limits. Firestore free tier: 50k reads, 20k writes, 20k deletes per day. With real-time listeners, reads accumulate fast. Use targeted queries (don't listen to entire collections). Batch writes where possible.

Cloud Function cold starts. First invocation after idle can take 3–8 seconds. If the revenue engine is a Cloud Function, the first round's simulation will feel slow. Pre-warm it before the game starts by calling it once.

Config as data, not code. Store all game parameters (coefficients, starting budget, product costs, etc.) in a Firestore config document, not hardcoded in Cloud Functions. This lets Game Design tune values without deploying code.
